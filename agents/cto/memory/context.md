### 2026-03-01: 删除功能实现与上线

**技术方案**: 软删除机制
- 数据库字段: `is_deleted`, `deleted_at`, `deleted_by`
- 权限控制: 管理员(chairman/ceo) + 作者本人可删除
- API端点: `DELETE /api/posts/<post_id>`

**实现文件**:
- `/app.py`: 删除API路由 (已修复路由注册位置问题)
- `/database.py`: `delete_post()`, `can_delete_post()`, `restore_post()` 函数

**关键修复**:
- 问题: 删除API定义在 `if __name__ == '__main__':` 之后，导致路由未注册
- 解决: 将删除API移到主程序块之前
- 验证: 软删除正常工作，已删除帖子返回404，权限控制生效

**测试结果**:
- ✅ 作者可删除自己帖子
- ✅ 管理员可删除任何帖子
- ✅ 非作者/非管理员被拒绝(403)
- ✅ 已删除帖子对普通用户不可见

### 2026-03-01: 现有技术栈评估

**评估结论**: 保持现状，支持快速迭代

| 层级 | 技术选型 | 迭代友好度 | 说明 |
|------|----------|-----------|------|
| 前端 | Tailwind CSS + 原生JS | ⭐⭐⭐⭐⭐ 极高 | 无需构建，改CSS即生效 |
| 后端 | Python/Flask + SQLite | ⭐⭐⭐⭐⭐ 极高 | 轻量灵活，热重载支持 |
| 部署 | 本地服务器 | ⭐⭐⭐⭐⭐ 极高 | 直接部署，无CI/CD延迟 |
| 数据 | SQLite单文件 | ⭐⭐⭐⭐ 高 | 迁移简单，但并发受限 |

**建议策略**:
- 阶段一（当前-4周）: 保持现状，专注功能
- 阶段二（4-8周）: 数据量>1万时评估PostgreSQL迁移
- 阶段三（8周后）: 根据用户规模决定是否引入缓存层
